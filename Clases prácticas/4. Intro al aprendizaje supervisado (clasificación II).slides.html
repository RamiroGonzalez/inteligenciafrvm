Pre-procesamiento de datos¶

Un problema muy frecuente (en industria es uno de los problemas m√°s reportados en ML) es que los datasets "crudos" no pueden recibir entrenamiento tal como est√°n, por lo que necesitan un pre-procesamiento para poder adaptarse a los requerimientos del modelo. Vemos algunos m√©todos de pre-procesamiento comunes:

Eliminación de features que no aportan a la predicción¶

Un problema com√∫n en los datasets es que muchas veces los mismos tienen algunas variables que no est√°n relacionadas de ninguna forma con las salidas o bien es sabido a priori que no aportan informaci√≥n significativa para predecir la salida.

  • Un ejemplo com√∫n son los identificadores, los cuales asocian cada fila de datos con un √∫nico c√≥digo de identificaci√≥n.
  • Estos predictores deben eliminarse, porque de lo contrario el modelo los ajustar√° e intentar√° hacer predicciones con datos que no tienen aporte alguno, empeorando la calidad general del modelo.

Ampliación o transformación del dataset¶

De forma similar, existen maneras de agregar o transformar datos en el dataset para mejorar las predicciones, por ejemplo incorporando informaci√≥n de datasets p√∫blicos o generando nuevos features a partir de los existentes o con la ayuda de informaci√≥n de dominio.

  • Este tipo de mejoras puede ayudar a mejorar las predicciones del dataset, por ejemplo al poder utilizar features m√°s relevantes para predecir las salidas.
  • Al igual que al eliminar los features (m√°s all√° de aquellos obvios como los identificadores), no hay una regla general para ampliar el dataset, pues depende mucho del dominio del problema.
  • Por ejemplo, en visi√≥n por computadoras un pre-procesamiento que se hace algunas veces consiste en pasar las im√°genes de RGB a escala de grises para reducir el tama√±o de la misma. Por otra parte, una ampliaci√≥n (data augmentation) com√∫n consiste en agregar la misma imagen pero rotada hacia la derecha e izquierda.

Dummy variables para datasets con features no numéricos¶

  • Por ejemplo, si tenemos que hacer una predicci√≥n sobre im√°genes geom√©tricas y contamos con una variable categ√≥rica "Color" en las mismas que toma valores "Rojo", "Azul" y "Verde". Tal variable no puede ser utilizada de esa forma para hacer la predicci√≥n, puesto que los modelos emplean features num√©ricos.
  • Una soluci√≥n r√°pida consistir√≠a en reemplazar los features asignando, por ejemplo, 0 cuando se trata de "Rojo" , 1 cuando se trata de "Azul" y 2 cuando es "Verde", es decir

  • Un enfoque que a veces se usa para estos casos consiste en crear un nuevo feature para cada valor categ√≥rico posible, asignando un 1 cuando el valor pertenece a la mencionada categor√≠a y un 0 para todas las dem√°s categor√≠as.

Desventajas:¶

One-hot encoding no es necesario, o puede causar un decremento de la performance cuando:

  • Usamos m√©todos basados en √°rboles como decision trees o random forest.
  • Tenemos s√≥lo dos valores posibles para la variable categ√≥rica.
  • Aumentar√≠a muy considerablemente la cantidad de dimensiones. En esos casos tenemos que evaluar otras opciones como crear features a mano que agrupen lo que queremos representar en una forma que tenga sentido en el dominio de los datos.

Normalización de los datos¶

En virtud de que todos los features contribuyan de forma equitativa (es decir que no tengan m√°s peso en la clasificaci√≥n aquellos features con mayor magnitud), para algunos modelos se requiere que los datos est√©n normalizados de alguna forma, por ejemplo para que todos los valores caigan en el mismo rango fijo. La normalizaci√≥n se hace en cada feature. Informalmente, "para cada columna".

  • Una normalizaci√≥n posible es minmax normalization, la cual transforma un feature de tal forma que todos sus valores caigan en el rango $[0,1]$.
  • Para lograr esto es suficiente hacer, para cada elemento $j$ del feature $X_i$
$$Z_{ij} = \frac{X_{ij} - min(X_i)}{max(X_i) - min(X_i)}$$

Anal√≠ticamente, $$Z_{ij} = \frac{X_{ij} - \bar{X}}{\sigma}$$ donde $\bar{X}$ es la media de la poblaci√≥n y $\sigma$ es la desviaci√≥n est√°ndar.
  • Cada valor $Z_{ij}$ normalizado representa la distancia entre el valor $X_{ij}$ y la media de la poblaci√≥n, en $Z_{ij}$ desviaciones est√°ndar como unidad.

A tener en cuenta en la normalización¶

1. Hay modelos que no son afectados por ciertas normalizaciones¶

Ejemplo: √°rboles de decisi√≥n o random forest.

2. ¿Qué sucede cuando queremos normalizar y tenemos que dividir los datos en train y test (o, análogamente, en train y validation)?¶

Si normalizamos toda nuestra matriz de datos X, tendremos un problema: estaremos sesgando indirectamente nuestro entrenamiento con respecto a los conjuntos de test y de validaci√≥n.

Lo que se hace en estos casos es:

  1. Obtener $\bar{X}_{train}$ y $\bar{\sigma}_{train}$ para el caso de normalizaci√≥n z-score, o $min(X_i)_{train}$ y $max(X_i)_{train}$ para el caso de minmax.
  1. Aplicar la normalizaci√≥n al conjunto de entrenamiento.
  1. Al hacer predicciones, normalizar el conjunto de validaci√≥n o test con respecto a los estad√≠sticos del conjunto de entrenamiento, es decir $\bar{X}_{train}$ y $\bar{\sigma}_{train}$ (para el caso de z-score). Es decir que para normalizar el conjunto de test, hacemos para cada uno de sus $X_{ij}$:
$$Z_{ij} = \frac{X_{ij} - \bar{X}_{train}}{\sigma_{train}}$$

Naturalmente, si hacemos esto perderemos alguna de las propiedades de la normalizaci√≥n para el conjunto de test (por ejemplo, en el caso de z-score, es muy probable que la media y desviaci√≥n de los datos de test difieran de 0 y 1, respectivamente), pero esto es totalmente preferible antes que sesgar el conjunto de entrenamiento.

Nota: ambas librer√≠as de ejemplo conservan los estad√≠sticos originales.

Tipos de errores¶

En la clasificaci√≥n binaria (es decir cuando la cantidad de clases posibles a la que una variable puede pertenecer es de dos), al intentar hacer una predicci√≥n, existen dos tipos de errores de predicci√≥n a considerar.

Por ejemplo, supongamos que testeamos la existencia de una enfermedad en un paciente. Como hip√≥tesis nula $h_0$ tomamos a "$h_0$: el paciente no posee la enfermedad", mientras que como hip√≥tesis alternativa tomamos "$h_1$: el paciente posee la enfermedad".

  • $h_0$ es la hip√≥tesis que se suele usar para establecer "ausencia o inexistencia de relaci√≥n entre dos fen√≥menos" (tomando como fen√≥menos "el paciente" y "la enfermedad"), mientras que $h_1$ representa "alg√∫n tipo de relaci√≥n entre dos fen√≥menos".
  • "Inocente hasta que se demuestre lo contrario". En testing de hip√≥tesis, $h_0$ es la postura por defecto. Esto quiere decir que para rechazarla y por lo tanto validar una hip√≥tesis alternativa debe demostrarse fehacientemente que $h_0$ es falsa m√°s all√° de toda duda razonable (normalmente representada por un intervalo de confianza del 95%).
  • El error de tipo I, false positive o falsa alarma ocurre cuando se rechaza incorrectamente una $h_0$ verdadera. En nuestro ejemplo, esto sucede cuando incorrectamente diagnosticamos la enfermedad en el paciente.
  • Por otra parte, el error de tipo II, false negative o detecci√≥n perdida, ocurre cuando incorrectamente se retiene una $h_0$ falsa, siendo que se daba la relaci√≥n establecida por $h_1$. En nuestro ejemplo esto sucede cuando detectamos incorrectamente que el paciente no registra la enfermedad.
  • En el caso de que la predicci√≥n haya sido correcta, decimos que la misma arroj√≥ un true positive o true negative, seg√∫n si se predijo correctamente la existencia o no de la relaci√≥n entre dos fen√≥menos (positive significa que dicha relaci√≥n existi√≥, es decir que se cumple $h_1$).
  • En nuestro caso, true positive representa la correcta predicci√≥n en la existencia de la enfermedad mientras que false positive representa correctamente la predicci√≥n sobre la ausencia de la enfermedad.
  • En nuestro ejemplo, si detect√°ramos varias enfermedades en el paciente y nos interesa saber $h_0$, true negative se referir√° a los casos donde predecimos que el paciente no posee ninguna enfermedad y tal situaci√≥n es la real; mientras que true positive se refiere a haber inferido correctamente que el paciente posee alguna enfermedad. Este enfoque se conoce como one-vs-all.

Concepto preliminar: en la clasificaci√≥n binaria, es decir aquellos problemas donde $c_i \in \{False, True\}$, el umbral o threshold $\theta \in [0,1]$ es la probabilidad m√≠nima que hace que el predictor clasifique una observaci√≥n como True. Aquellas probabilidades por debajo del umbral son clasificadas como False.

(nota: no confundir $\theta$ utilizado en el resto del notebook para denotar un umbral con $\theta$ com√∫nmente utilizado para referirse a los par√°metros del predictor)

Por ejemplo, para clasificar un positivo con un $\theta = 0.75$ se necesita que $P(y=1 \mid x) > 0.75$.

Métricas desde el punto de vista de los negativos¶

La primera m√©trica es el false positive rate (FPR) (o proporci√≥n de falsas alarmas), que representa el porcentaje de errores de TI en la clasificaci√≥n, es decir

$$ FPR = \frac{FP}{\text{actual negatives}} = \frac{FP}{FP + TN}$$

Un predictor no tendr√≠a falsas alarmas si clasificara todo como "False", al usar un umbral $\theta=1$.

Por otra parte, el false negative rate (FNR) (o proporci√≥n de detecciones perdidas) representa el porcentaje de errores TII en la clasificaci√≥n, es decir

$$ FNR = \frac{FN}{\text{actual positives}} = \frac{FN}{FN + TP}$$

Un predictor no tendr√≠a detecciones perdidas si clasificara todo como "True", usando un umbral $\theta = 0$.

Métricas desde el punto de vista de los positivos¶

Fuente: Walber (usuario de Wikipedia), imagen adaptada desde https://en.wikipedia.org/wiki/Precision_and_recall

Uno de los indicadores principales es el positive predictive value o precision, que representa el porcentaje de positivos correctamente inferidos como tal por cada positivo inferido. Est√° dado por

$$P(y=1 \mid \hat{y}=1) = \frac{TP}{\text{predicted positives}} = \frac{TP}{TP + FP}$$

Intuitivamente puede interpretarse como "qu√© tan seguro es un clasificador al inferir las observaciones como positivas". Un predictor maximizar√≠a esta m√©trica si clasificara como "True" s√≥lo aquellas observaciones con las que cuente con total certeza que son "True".

Otro indicador principal es el true positive rate, sensitivity o recall, que representa el porcentaje de positivos correctamente inferidos por cada positivo real, dado por

$$P(\hat{y}=1 \mid y=1)= \frac{TP}{\text{actual positives}} = \frac{TP}{TP + FN}$$

Intuitivamente puede interpretarse como "qu√© tan bueno es un clasificador en detectar los positivos". Un predictor ser√≠a perfecto en la detecci√≥n de positivos si clasificara todo como "True", al usar un umbral $\theta = 0$.

La primera medida que venimos usando, el porcentaje de aciertos (accuracy) est√° dada por

$$\text{accuracy} = \frac{\text{correctly predicted}}{\text{all predicted}} = \frac{TP + TN}{TP + FP + TN + FN}$$

Como es trivial y poco √∫til maximizar precision o recall por separado, un buen clasificador deber√≠a, en general (y dependiendo del problema que trata de resolver), intentar maximizar conjuntamente tanto precision como recall. Normalmente ambos representan intereses contrapuestos.

  • En el caso de que querramos comparar ambas m√©tricas con respecto a la otra en forma equitativa, para evaluarlas conjuntamente se suele usar el F1 score, dado por la media arm√≥nica
$$f_1 = \frac{2}{P^{-1}+R^{-1}}$$

False positive vs false negative tradeoff¶

Como acabamos de ver, la contraposici√≥n de intereses en un clasificador se da porque existe una relaci√≥n inversa entre los errores false positive (falsa alarma) y los errores false negative (detecci√≥n perdida).

  • La inclinaci√≥n que querramos darle a este tradeoff depende del problema que estemos intentando resolver.
  • Por ejemplo: imaginemos un clasificador en el dominio criminal, donde tenemos sospechosos y evidencias hacia ellos, y buscamos maximizar tanto que los sospechosos culpables sean identificados como tales (recall), como la seguridad de nuestro clasificador al afirmar culpabilidad de un sospechoso (precision), de modo de minimizar la cantidad de inocentes con condena injusta.
  • En este caso, hay posturas en filosof√≠a del derecho que realizan una clara distinci√≥n en este balance, como por ejemplo el ratio de Blackstone: "It is better that ten guilty persons escape than that one innocent suffer"
  • Para aquellas tareas que se desempe√±an en entornos con aversi√≥n al riesgo, por ejemplo al decidir si las pruebas contra un sospechoso son suficientes para afirmar su culpabilidad, debemos dise√±ar nuestro clasificador de tal forma que se minimicen los costos totales producidos por los errores de predicci√≥n.
  • En la clasificaci√≥n binaria, una de las opciones es incluir una reject region, es decir un umbral de probabilidad donde el predictor, en lugar de clasificar como False, no tomar√° decisiones, dejando las mismas en manos de una persona, por ejemplo.

Alternativamente, si queremos tomar esa decisi√≥n pero que nuestro clasificador sea conservador debido al posible costo de los errores, podemos usar el mismo umbral para definir hasta cu√°nto debe ser nuestro grado de seguridad para asegurar o rechazar $h_0$.

  • En los clasificadores probabil√≠sticos vistos aqu√≠, el umbral es establecido al ver la probabilidad de predicci√≥n de las distintas clases, y alterando la clase elegida cuando la misma no supera el umbral.
  • Por defecto, el umbral utilizado en los clasificadores binarios es $\theta=0.5$, mientras que en los clasificadores multi-clase se selecciona la clase para la cu√°l nuestro clasificador arroja mayor probabilidad.

Receiver operating characteristic (ROC) curve¶

  • Los umbrales tambi√©n pueden utilizarse para evaluar con qu√© margen de confianza nuestro clasificador infiere la clase de cada una de las observaciones, para de esta manera poder compararlo mejor frente a otros clasificadores en cuanto a su capacidad de generalizar.
  • Por ejemplo, el clasificador A y el clasificador B pueden tener id√©nticas predicciones para un dataset, pero A estar clasificando a todas las clases con un 95% de probabilidad y B estar clasificando con un 51%.
  • Distinguir entre ambos clasificadores es precisamente la idea de las curvas ROC. En ellas se muestra c√≥mo se comparan los distintos clasificadores en t√©rminos de TPR (recall) y FPR (falsas alarmas) con respecto a la suposici√≥n aleatoria (elige clase 0 o 1 aleatoriamente) y al clasificador perfecto, mostrando el equilibrio entre true positives y false positives para las distintas instancias.
  • Veamos c√≥mo se ve una curva ROC para nuestro ejemplo (armado a partir de https://github.com/reiinakano/scikit-plot).
